# GitHub Pages를 이용한 웹사이트 오픈(호스팅) 가이드

결론부터 말씀드리면, **현재 상태 그대로 GitHub Pages에 올릴 경우 "반쪽짜리"로만 작동**합니다.

---

## 1. 작동하는 것과 작동하지 않는 것

### 🟢 정상적으로 작동하는 부분 (Frontend)
- **화면 렌더링**: HTML, CSS, JavaScript를 이용한 시각적 디자인, 시계, 버튼 클릭 애니메이션 등 프론트엔드 요소는 완벽하게 뜹니다.
- GitHub Pages는 정적(Static) 파일 호스팅에 특화되어 있기 때문입니다.

### 🔴 작동하지 않는 부분 (Backend & Data)
- **직원 등록, 출퇴근 기록 저장, 데이터 불러오기 (불가)**
- 현재 시스템은 `server.js`(또는 IIS/Apache)를 통해 서버의 하드디스크에 있는 `members.json` 및 `allmember.json` 파일을 **직접 수정하고 덮어쓰는 방식**으로 데이터를 영구 저장합니다.
- 하지만 GitHub Pages는 보안 및 서버 구조상 **백엔드(Node.js, PHP 등) 실행을 지원하지 않으며, 파일 쓰기(POST 통신)를 허용하지 않습니다.**

---

## 2. GitHub Pages로 완벽하게 오픈하기 위한 대안

만약 반드시 GitHub Pages를 통해 무료로 홈페이지를 오픈하고 싶으시다면, **데이터 저장 방식(백엔드)을 수정**해야 합니다. 가장 추천하는 2가지 방법은 다음과 같습니다.

### 대안 A: 프론트엔드와 백엔드를 분리 호스팅 (추천)
가장 보편적인 방법으로, 화면(웹사이트)과 데이터 처리(서버)를 각각 무료 호스팅 서비스에 올리는 방법입니다.
1. **프론트엔드 (웹사이트 화면)**: GitHub Pages에 현재의 HTML, CSS, JS 파일들을 업로드합니다.
2. **백엔드 (데이터 저장소)**: 기존에 만들어둔 `server.js`와 JSON 파일들을 **Render, Railway, Heroku** 같은 무료 Node.js 호스팅 서비스에 업로드합니다.
3. **연동**: `script.js` 내부의 API 주소(`/api/members`)를 백엔드 호스팅 주소(예: `https://my-backend.onrender.com/api/members`)로 변경합니다.

### 대안 B: BaaS (BaaS: Backend as a Service) 사용 (서버 리스 방식)
`server.js`와 JSON 파일을 완전히 버리고, 구글의 **Firebase (Firestore)**나 **Supabase** 같은 클라우드 데이터베이스를 사용하는 방법입니다.
1. HTML, CSS, JS 파일에 Firebase SDK를 연동합니다.
2. 직원 등록, 출퇴근 버튼을 누르면 JSON 파일에 기록하는 대신, Firebase 데이터베이스에 실시간으로 기록되도록 `script.js`의 함수들을 수정합니다.
3. 이 경우 백엔드 서버 자체가 필요 없으므로 **GitHub Pages에 HTML 파일들만 올려도 완벽하게 전체 기능이 동작**합니다.

## 💡 요약
현재 만들어진 `test_web` 폴더를 그대로 GitHub에 올리시면 사이트 접속은 가능하지만, **출근 버튼을 누르거나 직원을 등록할 때 에러가 발생하며 데이터가 저장되지 않습니다.**
실제 운영을 위해 클라우드(인터넷)에 올리시려면 **대안 A(백엔드 분리)** 또는 **대안 B(Firebase 도입)** 중 하나를 선택하여 구조를 약간 수정해야 합니다!
